# TMS Test Deduplication

## Когда использовать
- Пользователь даёт xlsx-выгрузку из TestIT и просит найти дубли.
- Цель — отчёт по группам смысловых дубликатов с оценкой severity и отдельной категорией «подозрение на copy-paste шагов».

## Входные данные
xlsx-файл со структурой:
- колонка A — ID теста
- колонка C — название теста
- колонка F — шаг
- колонка H — ожидаемый результат
- строка с заполненными A и C = заголовок нового теста; последующие строки с пустыми A/C — шаги текущего теста.

## Ключевые идеи (прочти прежде чем запускать)

1. **Два сигнала отдельно**: `sim_title` и `sim_steps` считаются независимо. Если склеивать их в один текст, шаги всегда длиннее и подавляют сигнал названия — тесты с одинаковыми шагами, но разными названиями ошибочно слипаются.
2. **Disambiguator в названии = важный дискриминатор**. Пример:
   ```
   ППП-Т1213 Проверка поля карта отправителя (по счету)
   ППП-Т3215 Проверка поля карта отправителя (по карте)
   ```
   Если у двух тестов одинаковая «основа» названия (`stem`) и разные непустые уточнения в скобках / после тире / после двоеточия (`disambig`) — `sim_title` искусственно занижается до 0.2. Такая пара уйдёт в категорию `suspicious_copypaste`, а не в дубликаты. Это ровно кейс, когда тестировщик скопировал шаги и забыл адаптировать.
3. **Алгоритмические метки → LLM-верификация**:
   - `duplicate_full` / `duplicate_partial` / `same_intent_diff_flow` — LLM подтверждает severity и выбирает master
   - `suspicious_copypaste` — LLM по умолчанию ставит `severity=none, kind=copypaste_suspicion`; в отчёт уходит отдельной секцией «чинить шаги, не удалять»

## Пайплайн

Запускай скрипты строго по порядку из директории скилла. Все скрипты принимают `--help`.

### 1. Извлечение
```bash
python scripts/extract.py <input.xlsx> --out tests.json
```
Результат: `tests.json` вида `[{id, title, steps:[{action, expected}]}]`.

### 2. Двухсигнальная кластеризация
```bash
python scripts/cluster.py tests.json --out clusters.json --min-size 2
```
Результат: `clusters.json` — список групп кандидатов, у каждой:
- `label` ∈ {`duplicate_full`, `duplicate_partial`, `same_intent_diff_flow`, `suspicious_copypaste`}
- `test_ids` — члены группы (2..`--max-size`)
- `pair_metrics` — для каждой пары `sim_title`, `sim_steps`, `disambig_conflict`

Детерминированно, без LLM, работает на 2500+ тестах за минуты.

### 3. Верификация через LLM — **делает САМ АГЕНТ qwen**

`scripts/verify.py` не делает сетевых запросов к языковой модели. Это чистый оркестратор: он готовит промпты и потом агрегирует ответы. Всю работу с LLM выполняет агент qwen внутри qwen CLI.

Последовательность:

```bash
# шаг 3.1 — подготовка промптов
python scripts/verify.py clusters.json tests.json --out clusters_verified.json --batch-tokens 100000
```
После этого в `prompts_out/batch_NNN.txt` лежат готовые промпты. Каждый содержит:
- метку алгоритма и таблицу метрик пар
- тесты с шагами
- инструкцию с правилом disambiguator'а

Шаг 3.2 — для каждого `prompts_out/batch_NNN.txt` **ты, агент qwen**, должен:
1. Прочитать файл с промптом.
2. Сам обработать его своими силами (как LLM) — вернуть строгий JSON `{"results": [...]}` с полями `group_id`, `severity`, `kind`, `common_check`, `master_id`, `drop_ids`, `rationale`.
3. Записать получившийся JSON в `responses_out/batch_NNN.json` (путь строго соответствующий — `batch_000` → `batch_000.json`).

Шаг 3.3 — повторить запуск `verify.py` с теми же аргументами. Скрипт перечитает `responses_out/` и агрегирует ответы в `clusters_verified.json`. Если каких-то батчей не хватает — скрипт напишет, сколько осталось; просто досчитай их и запусти ещё раз.

### 4. Отчёт
```bash
python scripts/report.py clusters_verified.json tests.json --out report.md
```
Открой `report.md` и покажи пользователю summary. В отчёте 4 раздела (по `label`), в каждой группе — вердикт LLM (severity + kind) и таблица метрик пар.

## Важно

- Никогда не пытайся скормить все 2500 тестов в LLM одним промптом — не влезет и не нужно.
- Этапы 1–2 детерминированные, кэшируй результаты.
- Группы в категории `suspicious_copypaste` **не предлагать к удалению** — это обычно баг автора теста; рекомендация всегда «починить шаги».
- Счётчик «тестов к удалению» в summary учитывает только группы с `kind=duplicate`.
- Если пользователь хочет повысить recall (ловить более тонкие перефразы) — предложи установить `pymorphy3` (русская лемматизация) и/или переключить эмбеддер на `sbert` (см. `scripts/embedders/`).
